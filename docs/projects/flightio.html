Flightio

Internship Project

#java #android #flylitchi #drone #mapping #photogrammetry

Planning a flight for photogrammetry can be a time consuming task. There are some online tools, like flylitchi, that allow you to manage and easily create your plans - but with limited options.

Flightio is an android application made in Java for drone flight mapping, used for remote sensing and photogrammetry. This application export a ready-to-go plan regarding the specific camera settings, and export it to your flylitchi account and drone.

This project was made at the IGN center within a bigger project for multispectral photogrammetry.

When : 07.2019
Where : Forcalquier, France
Duration : 3 days

Context

At first, we were creating manually a flight plan on flylitchi. But this simple task took us more time than expected, as we needed to constantly adapt the initial plan to the terrain. It can be difficult to estimate precisely a flight / altitude from only satellites images - or 2D images in genral.

I took the initiative to create this mobile application, so we could quickly adapt a flight to the terrain. As we were working on multiple sites in the Alps of High Provence - and sometimes in rough conditions, it was easier to modify it through a mobile phone than a laptop.

Aerial photogrammetry

Introduction

Photogrammetry is the science of computing measurements from photographs. This technology is used to generate 3D objects from a set of images. We used MicMac to generate multispectral 3D maps from our aerial images, but this is another project.

Flightio was used to generate a flight plan and export it to our flylitchi account and device. The purpose of this application was to automate this task and to guarantee the respect of the coverage between photos - a constraint of photogrammetry. The more the images overlap, the more points will make up the 3D model.

Constraints

The camera's manufacturer often specify the captor size \(l\) and the size of a photosite \(p\) - i.e a pixel, or the focal length \(f\). From this data, we can estimate the altitude \(H\) and flight speed to make sure that our covering \(C\) is respected.
Let's visualize what I mean:

A bunch of mathematical equations are used to compute one variable from anoter.

The number of photosites in the axis \(i\) is \(n_i = l_i / p_i\),
The ground pixel size is \(P_i = H * p_i / f\),
The ground image size is \(L_i = P_i * n_i\),
The base between two photos is \(B_i = L_i * (1 - C_i)\), etc.

Once the base and altitude are computed, it's start to generate a plan that respect these constraints.

Mapping

The creation of the plan is quite simple. The user specify a bbox where the drone will fly over. Then a succession of interpolations are executed to generated Points of Interest (POI), where the drone will shoot a photo.

A minimal layout

Keep it simple

In only three days, I did't had time to make a complex application. In addition, I was working on this project at the same time, so I made a basic UI and minimal layout that satisfy our needs. I coded two Fragment Pages, one for the settings and the other for mapping. Both pages are linked, so that the settings computed from the first Fragment are send to the second.

Elegantly easy

The user only needs to set the camera used and its covering / altitude to generate a plan. Then, the user swipe to the second view and draw the contour of its flight. The generate button is used to create the drone plan.
